# Architectural Design: Network Simulator

Цей документ описує загальну архітектуру застосунку для моделювання обчислювальних систем та комп'ютерних мереж. Програму спроєктовано з використанням класичних патернів GoF (Creational, Structural, Behavioral) для забезпечення гнучкості, масштабованості та зручності тестування.

## Використані патерни проєктування

Нижче наведено опис 12 патернів проєктування, які імплементовано в архітектурі системи.

### Породжувальні (Creational)
1. **Singleton (`SimulationManager`)**
   * **Проблема:** У системі має бути лише одна точка керування часом, подіями та життєвим циклом симуляції.
   * **Рішення:** Забезпечує існування лише одного екземпляра менеджера та надає глобальний доступ до нього.
2. **Builder (`NetworkBuilder`)**
   * **Проблема:** Створення складної мережі (з маршрутизаторами, серверами та зв'язками) вимагає складного конструктора з багатьма параметрами.
   * **Рішення:** Дозволяє конструювати мережу покроково (додати сервер, додати підмережу) та генерувати кінцевий об'єкт `Subnet`.
3. **Prototype (`TopologyPrototype`, `StarTopology`)**
   * **Проблема:** Необхідність швидко створювати типові конфігурації мереж (наприклад, топологію "Зірка") без ручного налаштування кожного вузла.
   * **Рішення:** Надає інтерфейс для клонування готових заготовок мереж.

### Структурні (Structural)
4. **Composite (`NetworkNode`, `ServerNode`, `Subnet`)**
   * **Проблема:** Мережа має ієрархічну структуру: глобальна мережа складається з підмереж, які складаються з окремих серверів.
   * **Рішення:** Дозволяє клієнтам однаково працювати як з окремим сервером, так і з цілою підмережею через спільний інтерфейс `processTraffic()`.
5. **Decorator (`NodeDecorator`, `FirewallDecorator`)**
   * **Проблема:** Необхідність додавати новий функціонал (наприклад, фільтрацію трафіку фаєрволом) до існуючих вузлів без зміни їхнього коду.
   * **Рішення:** Обгортає `NetworkNode` у декоратор, який виконує перевірку пакетів перед передачею їх до базового вузла.
6. **Facade (`ReportFacade`)**
   * **Проблема:** Збір метрик та генерація звітів вимагає взаємодії з багатьма різними підсистемами симулятора.
   * **Рішення:** Надає простий єдиний інтерфейс для виведення фінального результату моделювання.

### Поведінкові (Behavioral)
7. **Strategy (`RoutingStrategy`, `ShortestPathStrategy`)**
   * **Проблема:** Сервери можуть використовувати різні алгоритми маршрутизації пакетів, які потрібно змінювати під час работы програми.
   * **Рішення:** Інкапсулює алгоритми маршрутизації в окремі класи, дозволяючи підміняти їх на льоту.
8. **State (`NodeState`, `OnlineState`, `OfflineState`)**
   * **Проблема:** Поведінка сервера кардинально змінюється залежно від того, чи він працює нормально, чи відключений, чи перевантажений.
   * **Рішення:** Виносить логіку поведінки для кожного стану в окремий клас.
9. **Command (`SimulationCommand`, `StartCommand`)**
   * **Проблема:** Графічний інтерфейс має відправляти команди ядру симуляції, але ці підсистеми повинні бути слабко пов'язані.
   * **Рішення:** Інкапсулює запити (старт, пауза) як об'єкти, що дозволяє легко додавати нові дії керування з UI.
10. **Observer (`SimulationObserver`, `MetricsCollector`)**
    * **Проблема:** Графічний інтерфейс та збирач метрик повинні миттєво оновлюватися, коли стан мережі змінюється.
    * **Рішення:** Менеджер симуляції сповіщає всіх "підписників" про зміни без жорсткої прив'язки до конкретних класів UI.
11. **Iterator (`NodeIterator`)**
    * **Проблема:** Необхідність послідовно обійти всі вузли в складній деревоподібній структурі `Subnet` для їх оновлення.
    * **Рішення:** Надає універсальний механізм обходу вузлів, приховуючи внутрішню структуру зберігання списків.
12. **Memento (`SimulationMemento`)**
    * **Проблема:** Необхідність зберегти поточний стан симуляції (пакети, стани серверів), щоб реалізувати функцію "Пауза/Відновлення" або збереження у файл.
    * **Рішення:** Дозволяє зробити "знімок" стану `SimulationManager` без порушення інкапсуляції.
