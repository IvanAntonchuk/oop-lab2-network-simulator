# Architectural Design: Network Simulator

Цей документ описує загальну архітектуру застосунку для моделювання обчислювальних систем та комп'ютерних мереж. Програму спроєктовано з використанням класичних патернів GoF (Creational, Structural, Behavioral) для забезпечення гнучкості, масштабованості та зручності тестування.

## Використані патерни проєктування

Нижче наведено опис 12 патернів проєктування, які імплементовано в архітектурі системи.

### Породжувальні (Creational)
1. **Singleton (`SimulationManager`)**
   * **Проблема:** У системі має бути лише одна глобальна точка доступу до керування історією операцій (Undo/Redo) та розсилкою сповіщень про події.
   * **Рішення:** Забезпечує існування лише одного екземпляра менеджера та надає доступ до нього з будь-якої точки програми.
2. **Builder (`NetworkBuilder`)**
   * **Проблема:** Створення складної мережі вимагає покрокового конструювання з багатьма параметрами.
   * **Рішення:** Дозволяє будувати мережу крок за кроком (додавання серверів, підмереж) або генерувати готові складні топології єдиним викликом (метод `buildStarTopology`), повертаючи кінцевий об'єкт `Subnet`.
3. **Prototype (`TopologyPrototype`, `NetworkNode::clone`)**
   * **Проблема:** Необхідність швидко створювати точні копії виділених частин мережі (графів) зі збереженням їхніх індивідуальних налаштувань (статуси, фаєрволи, алгоритми маршрутизації).
   * **Рішення:** Надає інтерфейс `clone()` для глибокого копіювання існуючих логічних вузлів без прив'язки до їхніх конкретних класів.

### Структурні (Structural)
4. **Composite (`NetworkNode`, `ServerNode`, `Subnet`)**
   * **Проблема:** Мережа має ієрархічну структуру: глобальна мережа складається з підмереж, які, у свою чергу, містять окремі сервери.
   * **Рішення:** Дозволяє клієнтам та системі однаково працювати як з окремим вузлом, так і з цілою підмережею через загальний базовий інтерфейс.
5. **Decorator (`NodeDecorator`, `FirewallDecorator`)**
   * **Проблема:** Необхідність динамічно додавати новий функціонал (наприклад, фільтрацію трафіку фаєрволом) до існуючих вузлів без спадкування та зміни базового коду.
   * **Рішення:** Обгортає `NetworkNode` у декоратор, який може розширювати логіку обробки трафіку перед передачею виконання базовому вузлу.
6. **Facade (`ReportFacade`)**
   * **Проблема:** Запуск діагностики мережі, пошук роутерів, обчислення всіх шляхів та збір логів вимагає складної взаємодії з багатьма підсистемами ядра.
   * **Рішення:** Надає простий єдиний інтерфейс для графічного вікна (UI), приховуючи під капотом усю складність математичних обчислень та генерації фінального звіту.

### Поведінкові (Behavioral)
7. **Strategy (`RoutingStrategy`, `ShortestPathStrategy`, `RandomPathStrategy`)**
   * **Проблема:** Сервери можуть використовувати різні алгоритми маршрутизації (пошук в ширину BFS або рандомізований пошук в глибину DFS), які призначаються користувачем динамічно.
   * **Рішення:** Інкапсулює алгоритми маршрутизації в окремі класи, дозволяючи легко перемикати логіку пошуку шляху для кожного вузла під час виконання програми.
8. **State (`NodeState`, `OnlineState`, `OfflineState`)**
   * **Проблема:** Поведінка сервера кардинально змінюється залежно від його поточного стану (увімкнений/вимкнений), що впливає на його здатність приймати трафік.
   * **Рішення:** Виносить логіку поведінки у відповідні класи-стани. Наприклад, вузол у стані `OfflineState` автоматично ігнорується алгоритмами маршрутизації.
9. **Command (`SimulationCommand`, `AddNodeCommand`, `DeleteNodeCommand` та ін.)**
   * **Проблема:** Усі дії користувача в графічному редакторі (додавання, видалення, з'єднання вузлів) потрібно відстежувати для можливості їх скасування (`Ctrl+Z`).
   * **Рішення:** Перетворює кожен запит або операцію на самостійний об'єкт-команду, який містить логіку виконання (`execute()`) та зворотного відкату (`undo()`).
10. **Observer (`SimulationObserver`, `DashboardObserver`)**
    * **Проблема:** Графічні віджети (наприклад, панель живої статистики) повинні миттєво оновлюватися при будь-яких змінах на сцені, але не повинні бути жорстко прив'язані до ядра мережі.
    * **Рішення:** Менеджер симуляції (Видавець) автоматично розсилає сигнал `NETWORK_CHANGED` усім підписникам, а `DashboardObserver` перехоплює його і оновлює дані на екрані.
11. **Iterator (`NodeIterator`, `SubnetIterator`)**
    * **Проблема:** Для оновлення інформації на дашборді необхідно обійти всі вузли деревоподібної мережі (Composite), не розкриваючи її внутрішньої реалізації.
    * **Рішення:** Надає безпечний механізм (Ітератор) для послідовного доступу до кожного елемента підмережі `Subnet`.
12. **Memento (`SimulationMemento`)**
    * **Проблема:** Для коректної роботи системи скасування дій (Undo/Redo) необхідно зберігати та відновлювати "знімки" (snapshots) стану логічних мап та структур мережі.
    * **Рішення:** Дозволяє зберігати поточний стан зв'язків та об'єктів симуляції у спеціальний об'єкт-зберігач і відновлювати його без порушення інкапсуляції класів ядра.
