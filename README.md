# Network Simulator

**Лабораторна робота №2 з ООП**

Цей проєкт — інтерактивний симулятор комп'ютерних мереж, розроблений мовою **C++17** з використанням фреймворку **Qt6** для візуалізації графічного інтерфейсу та **GTest** для юніт-тестування.

* **Варіант:** 3
* **Підваріант:** Моделювання обчислювальних систем та комп'ютерних мереж
* **Документація API (Doxygen):** [Читати документацію](https://ivanantonchuk.github.io/oop-lab2-network-simulator/)

---

## Основний функціонал програми

* **Візуальний редактор топологій:** Додавання серверів та маршрутизаторів на робочу область (QGraphicsScene), створення та видалення з'єднань між ними за допомогою миші.
* **Керування станами вузлів:** Можливість "вимкнути" сервер (Offline) або увімкнути йому захист (Firewall), що миттєво впливає на маршрутизацію.
* **Розумна маршрутизація:** Пошук найкоротшого шляху (Shortest Path) або випадкового безпечного маршруту (Randomized DFS) між вузлами з урахуванням обривів мережі та вимкнених серверів.
* **Undo / Redo система:** Повне збереження історії дій користувача. Будь-яке додавання, видалення, з'єднання або зміну стану можна скасувати та повернути назад.
* **Клонування мереж:** Можливість миттєво створити глибоку копію існуючої мережі (або шаблону "Зірка").
* **Живий дашборд та звіти:** Моніторинг активності мережі в реальному часі та генерація детальних текстових звітів про стан кожного вузла.

## Реалізовані Патерни Проєктування (GoF)

У проєкті успішно інтегровано **12 класичних патернів проєктування** для забезпечення гнучкості та масштабованості коду.

### Породжувальні (Creational)
| Патерн | Клас-реалізація | Опис використання |
| :--- | :--- | :--- |
| **Singleton** | `SimulationManager` | Глобальна точка доступу до симуляції, яка гарантує існування лише одного екземпляра менеджера подій та історії. |
| **Builder** | `NetworkBuilder` | Покрокове створення складних конфігурацій підмереж (наприклад, автоматична побудова топології "Зірка"). |
| **Prototype** | `TopologyPrototype` | Механізм глибокого ізольованого клонування існуючих мережевих вузлів та цілих топологій. |

### Структурні (Structural)
| Патерн | Клас-реалізація | Опис використання |
| :--- | :--- | :--- |
| **Composite** | `Subnet`, `NetworkNode` | Побудова деревоподібної ієрархії, де підмережа може містити як окремі сервери, так і інші підмережі, обробляючи їх однаково. |
| **Decorator** | `FirewallDecorator` | Динамічне додавання поведінки до сервера (фільтрація пакетів) без зміни його базового класу. |
| **Facade** | `ReportFacade` | Спрощений єдиний інтерфейс для збору діагностики та генерації комплексного звіту про стан усієї мережі. |

### Поведінкові (Behavioral)
| Патерн | Клас-реалізація | Опис використання |
| :--- | :--- | :--- |
| **Strategy** | `RoutingStrategy` | Взаємозамінні алгоритми пошуку шляху (`ShortestPathStrategy`, `RandomPathStrategy`), які можна змінювати на льоту. |
| **State** | `NodeState` | Зміна поведінки вузла в залежності від його стану (`OnlineState`, `OfflineState`), що автоматично впливає на обробку трафіку. |
| **Command** | `ICommand` | Інкапсуляція дій користувача (додавання, з'єднання, видалення) в об'єкти для реалізації Undo/Redo. |
| **Memento** | `SimulationMemento` | Збереження знімків стану мережі для безпечного відкату змін у `HistoryManager`. |
| **Observer** | `SimulationObserver` | Автоматичне сповіщення графічного інтерфейсу (`DashboardObserver`) про будь-які структурні зміни в логічному ядрі. |
| **Iterator** | `SubnetIterator` | Безпечний послідовний обхід складної деревоподібної структури Композита без розкриття її внутрішньої реалізації. |

## Тестування (GTest)

Проєкт покритий автоматизованими юніт-тестами з використанням **Google Test**. 
Написано **24 тести**, які перевіряють:
1. Коректність роботи кожного з 12 патернів проєктування.
2. Стійкість ядра до граничних випадків (Edge Cases): обробка розірваних мереж, циклічних маршрутів, ігнорування Offline-вузлів.
3. Безпеку роботи з пам'яттю (Memory Safety) та відсутність витоків під час масового створення/видалення об'єктів.

## Розширений звіт щодо використання патернів проєктування

У цьому розділі наведено детальний аналіз 12 реалізованих патернів проєктування (GoF), обґрунтування їх використання, а також переваги та недоліки в контексті симулятора комп'ютерних мереж.

---

### 1. Породжувальні патерни (Creational Patterns)

#### 1.1. Singleton (Одинак)
* **Де використано:** `include/core/SimulationManager.h`, `src/core/SimulationManager.cpp`.
* **Яку проблему вирішує:** Забезпечує існування лише одного екземпляра менеджера симуляції, який керує глобальними подіями мережі та історією команд.
* **Переваги:** Зручний глобальний доступ до `HistoryManager` та списку `SimulationObserver` з будь-якого місця програми (особливо з UI команд).
* **Недоліки:** Створює глобальний стан, що ускладнює незалежне юніт-тестування (для уникнення цього в тестах використовується явна передача вказівника на менеджер).

#### 1.2. Builder (Будівельник)
* **Де використано:** `include/patterns/creational/NetworkBuilder.h`, `src/patterns/creational/NetworkBuilder.cpp`.
* **Яку проблему вирішує:** Ізолює складний процес створення топологій (наприклад, метод `buildStarTopology`) від їхнього представлення. Дозволяє покроково додавати сервери та маршрутизатори до об'єкта `Subnet`.
* **Переваги:** Код створення великих мереж стає читабельним; можливість конструювати різні топології одним набором методів.
* **Недоліки:** Вимагає створення окремого класу будівельника, що додає надмірності для створення простих одиночних вузлів.

#### 1.3. Prototype (Прототип)
* **Де використано:** `include/network/NetworkNode.h` (інтерфейс `clone()`), `include/patterns/creational/TopologyPrototype.h`, `StarTopology.h`.
* **Яку проблему вирішує:** Дозволяє системі (особливо при натисканні кнопки "Clone Network") створювати глибокі копії об'єктів без прив'язки до їхніх конкретних класів.
* **Переваги:** Легке клонування складних `Subnet`, яке ізолює оригінальну мережу від модифікацій у її копії (що доведено юніт-тестами).
* **Недоліки:** Складність реалізації глибокого копіювання (Deep Copy) через наявність циклічних посилань між вузлами мережі (вирішено через обережне копіювання векторів `std::weak_ptr`).

### 2. Структурні патерни (Structural Patterns)

#### 2.1. Composite (Компонувальник)
* **Де використано:** `include/network/NetworkNode.h` (Component), `ServerNode.h` (Leaf), `Subnet.h` (Composite).
* **Яку проблему вирішує:** Дозволяє клієнтам (наприклад, алгоритмам маршрутизації або `ReportFacade`) однаково обробляти як окремий `ServerNode`, так і цілу `Subnet` з сотнею вузлів через єдиний метод `processTraffic()`.
* **Переваги:** Максимально спрощує додавання нових типів вузлів і формування складної ієрархічної структури локальних мереж.
* **Недоліки:** Дизайн стає занадто загальним — важко обмежити, які саме компоненти можуть бути додані до `Subnet` на рівні системи типів C++.

#### 2.2. Decorator (Декоратор)
* **Де використано:** `include/patterns/structural/NodeDecorator.h`, `FirewallDecorator.h`.
* **Яку проблему вирішує:** Динамічно додає нову поведінку (перевірку безпеки пакетів / Firewall) до існуючих вузлів без створення підкласів на кшталт `SecureServerNode`.
* **Переваги:** Забезпечує дотримання принципу Single Responsibility. Фільтрацію трафіку можна "надягнути" та "зняти" з вузла під час виконання програми.
* **Недоліки:** Призводить до появи великої кількості дрібних об'єктів-обгорток, які ускладнюють ідентифікацію базового вузла при відлагодженні.

#### 2.3. Facade (Фасад)
* **Де використано:** `include/patterns/structural/ReportFacade.h`, `src/patterns/structural/ReportFacade.cpp`.
* **Яку проблему вирішує:** Ховає складність збору статистики (прохід по всім ітераторам, перевірка станів вузлів, агрегація шляхів) за одним простим методом `generateNetworkReport()`.
* **Переваги:** Клієнтський код (клас `MainWindow`) взагалі нічого не знає про внутрішню структуру мережі при виведенні звітів.
* **Недоліки:** Існує ризик, що фасад перетвориться на "God Object", якщо в нього постійно додаватимуться нові бізнес-вимоги до аналітики.

### 3. Поведінкові патерни (Behavioral Patterns)

#### 3.1. Strategy (Стратегія)
* **Де використано:** `include/patterns/behavioral/RoutingStrategy.h`, `ShortestPathStrategy.h`, `RandomPathStrategy.h`, виклик у `ServerNode::setStrategy()`.
* **Яку проблему вирішує:** Дозволяє динамічно змінювати алгоритм пошуку шляху (наприклад, з OSPF/BFS на рандомізований DFS) під час роботи симулятора без зміни коду самих серверів.
* **Переваги:** Дотримання Open/Closed Principle. Додати новий алгоритм маршрутизації можна шляхом створення одного нового класу.
* **Недоліки:** Клієнт (UI або користувач) повинен розуміти різницю між стратегіями, щоб вибрати правильну.

#### 3.2. State (Стан)
* **Де використано:** `include/patterns/behavioral/NodeState.h`, `OnlineState.h`, `OfflineState.h`. Зміна стану викликається у `ServerNode::changeState()`.
* **Яку проблему вирішує:** Виключає необхідність писати масивні конструкції `if (isOffline) { ... } else { ... }` в методах обробки трафіку вузла.
* **Переваги:** Логіка обробки трафіку та участі в маршрутизації зосереджена у відповідних класах станів. Легко додавати нові стани (наприклад, `MaintenanceState`).
* **Недоліки:** Кількість класів у проєкті зростає. Стани мають знати про контекст (`ServerNode`), що створює перехресну залежність.

#### 3.3. Command (Команда)
* **Де використано:** `include/patterns/behavioral/SimulationCommand.h`, та конкретні команди `AddNodeCommand.h`, `ConnectCommand.h`, `DeleteNodeCommand.h`, `ToggleStateCommand.h`.
* **Яку проблему вирішує:** Інкапсулює запит (наприклад, дію користувача мишкою) як об'єкт, дозволяючи зберігати їх у чергу або стек для системи Undo/Redo.
* **Переваги:** Повне від'єднання графічного інтерфейсу (Invoker) від логічного ядра (Receiver). Ідеальна реалізація багатоступеневого скасування дій.
* **Недоліки:** На кожну кнопку чи дію на сцені доводиться створювати окремий клас команди, що суттєво збільшує обсяг коду.

#### 3.4. Memento (Зберігач)
* **Де використано:** `include/patterns/behavioral/SimulationMemento.h`. Взаємодіє з `HistoryManager`.
* **Яку проблему вирішує:** Зберігає "знімок" (snapshot) поточного стану мережі перед складними деструктивними діями (видаленням великих підмереж), щоб відновити їх при Undo.
* **Переваги:** Не порушує інкапсуляцію ядра — внутрішній стан об'єктів зберігається і відновлюється без надання прямого доступу до приватних змінних.
* **Недоліки:** Якщо мережа налічує тисячі вузлів (як у проведеному стрес-тесті), збереження знімків вимагатиме значних витрат оперативної пам'яті.

#### 3.5. Observer (Спостерігач)
* **Де використано:** `include/patterns/behavioral/SimulationObserver.h`, `DashboardObserver.h`, `SimulationManager::notifyObservers()`.
* **Яку проблему вирішує:** Механізм підписки, який повідомляє графічний дашборд про те, що топологія мережі змінилася і статистику потрібно перерахувати.
* **Переваги:** Ослаблене зв'язування (Loose Coupling). Ядро нічого не знає про існування Qt-віджетів, воно просто розсилає сигнал `event`.
* **Недоліки:** Якщо в мережі відбувається багато дрібних подій (наприклад, масове підключення сотень кабелів), каскадні оновлення дашборду можуть знизити продуктивність.

#### 3.6. Iterator (Ітератор)
* **Де використано:** `include/patterns/behavioral/NodeIterator.h`, `SubnetIterator.h`, виклик у `Subnet::createIterator()`.
* **Яку проблему вирішує:** Надає універсальний спосіб послідовного доступу до вузлів складної мережі (Composite), не розкриваючи того, що вони зберігаються у векторі `std::vector<std::shared_ptr<NetworkNode>>`.
* **Переваги:** Підтримує принцип єдиної відповідальності: логіка обходу ієрархії винесена в окремий клас ітератора.
* **Недоліки:** Для найпростіших мереж (де всі вузли лежать у плоскому масиві) використання стандартних ітераторів C++ (`std::vector::iterator`) було б більш легковагим рішенням.
